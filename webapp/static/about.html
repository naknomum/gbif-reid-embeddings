Overview

This is a suite of tools for analyzing wildlife photo data from GBIF occurrences, primarily for the purpose of identifying specific individual animals across all photo-based occurrences of a given species, e.g. for mark-recapture re-ID.

This is accomplished by creating a very simple database of vector embeddings for each animal in a photo. In a way this can be thought of as a companion database to the photos and occurrences in GBIF. One image may have multiple animals, each of which is represented by a rectangular set of pixels called an annotation. From each annotation is extracted an embedding vector, which acts as a sort of fingerprint tied to the visual uniqueness of that animal. The more similar two embeddings are (shorter distance in the vector space), the more likely the corresponding annotations represent the same animal.

Data structure

Data is stored in simple tables in pgvector, which is a vector-enabled extension to PostgreSQL. The tables only represent new data extracted from the GBIF-sourced image data, and point back to the original GBIF occurrence as reference. The tables can be thought of as a layer of metadata on top of each image that lives in GBIF.

The tables used are:

  1. image
    • id - primary key
    • uri - uri of image (from GBIF api)
    • gbif_id - reference back to GBIF occurrence
  2. annotation
    • id - primary key
    • image_id - foreign key to the image that this annotation is associated with
    • bbox - data about rectangle area on the image where this annotation is positioned
    • embedding - a vector representing the embedding
Other than the embedding vector, this is a very basic, familiar database schema. Searching for similarity matches across multiple annotations, based on vector distance, is very fast in pgvector, so this lightweight setup is suited to handle very large volumes of data. Image data storage is temporary, as it is only needed at the time of embedding extraction; so the size of the data is also relatively compact.

Current Implementation

The current prototype runs as a set of docker containers using docker compose. There are three containers used:

  • pgvector - initialized with empty schema as described above
  • python web app - a very simple demo to select a "query annotation" and find matching annotations in the database
  • ml-service - the container which handles both the detection of annotations in the image (finding the bounding boxes) and extracting the embedding vector from those annotation boxes

ml-service details

At present in the alpha development stage, the ml-service container requires a bit more hands-on assistance to get running. Since it is only needed occasionally (when new image data is first ingested), the hopes for future development is to have this container spin up when needed. Once the annotations and embeddings are extracted, only the pgvector and webapp containers need be run for user queries.

For high volumes of image data, ideally the ml-service container would have access to a GPU, but is capable of running on CPU only machines for small-to-medium sized data ingestion.

You can find more information about ml-service on github. Specifically, the prototype uses the pipeline feature of ml-service, which supports detecting all annotations in the image while simultaneous providing the extracted embedding vectors for each, all via a single api call.

The documentation lists the numerous models and options supported, which can be tailored to specific species or image data as needed. In the case of this prototype, for the embedding extraction, the MiewID model is being used; specifically MiewID v4.1, which supports dozens of species with a high degree of matching accuracy. It produces a 2152-length vector, which the pgvector table is initialized to support.

Demo

A simple demo is provided to help visualize vector matching results. This demo runs on a very small set of data which represents only 300 Grevy's zebra (Equus grevyi) occurrences, pulled somewhat arbitrarily from GBIF via an api search based on taxonomy and existence of photos.

Despite the small sample size, the demo still manages to reveal numerous matches of zebra across unrelated occurrences spread over time and geographical space. Some interesting examples to explore, which include matches with high (> ~0.6) scores across diverse user submissions, matches on front/oblique orientations, etc: 369, 388, 473, 508, 542.

Future development

Currently, data is pulled from GBIF manually via a small script, then processed by ml-service to populate the pgvector database. Potential enhancements could include:

  • allowing users to do on-the-fly processing of queries to GBIF
  • nightly updates of embeddings for newly added GBIF occurrences
  • annotations/embeddings extracted from local images (not in GBIF) and matched against GBIF embeddings
  • open data sharing, such as COCO export for inclusion in lila.science, or for use as training data, etc.
  • development of GBIF companion metadata api: input GBIF occurrence id, return annotations/embeddings/matches
